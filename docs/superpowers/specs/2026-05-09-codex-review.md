## 1. Correctness Of Event-Sourcing Patterns

The top-level CQRS shape is correct: commands carry intent, aggregates emit immutable facts, projections are read models, and readers avoid aggregates directly (spec lines 79-84). The aggregate list also maps cleanly to bounded lifecycle concepts (lines 86-96). The first correction I would require before implementation is to keep aggregates pure. `ProjectAggregate.handle()` is shown handling `FetchPaperContent` and `ExtractMetadata` directly (lines 360-373), while external HTTP and parsing are adapter work. Those commands should be handled by application services/process managers that append `PaperFetchRequested` or `MetadataExtractionRequested`; adapters then append `PaperFetched`/`PaperFetchFailed` and `MetadataExtracted`. Aggregates should validate state transitions, not perform IO.

The process-manager model is conceptually right but underspecified. `IndexingFlow` joins `ParsingCompleted` and `DiscoveryCompleted` before dispatching `StartIndexing` (lines 638-645), but `react()` has no durable inbox, outbox, or handled-event record (lines 621-628). Add a manager inbox table keyed by `(manager_name, event_id)` and a transactional outbox for emitted commands. Without that, replay or crash recovery can double-dispatch or lose `StartIndexing`.

Captured-response replay is necessary but not sufficient. The event only stores URL, method, headers, status, body hash/path, and duration (lines 299-306). It must also capture request body hash/path, response headers, redirect chain, network error/timeout shape, retry attempt number, and a replay cache key. More importantly, the PRD makes `web_search()`, `notebook_query()`, `rlm_query()`, and Chroma semantic discovery part of the context workspace (PRD lines 1167-1177, 1582-1602). Those are external or model-dependent surfaces too. Treat every LLM call, embedding call, web search, and NotebookLM query as captured external interactions, not just raw HTTP through `httpx` (spec lines 49-50).

## 2. Scalability Ceilings

The SQLite claim is too optimistic. The spec says WAL plus `synchronous=NORMAL` gives about 10K writes/sec and that the single-writer model fits CQRS naturally (lines 251-256), but this workload emits many small events: sections/references/figures are individual events (line 422), chunks and embeddings are individual events (lines 470-479), and every external response is captured (lines 294-309). With JSON serialization, blob fsyncs, projection updates, and Chroma writes, the practical ceiling is more likely hundreds to low thousands of durable appends/sec on developer machines, lower under mixed read/projection load.

Use SQLite while the system is single-node, single-user or low-concurrency, and below roughly one active ingestion/indexing pipeline per process. Move when any of these happen: multiple projects ingest concurrently, remote runners arrive in Phase 3 (PRD lines 2482-2496), multi-user projects arrive in Phase 4 (PRD lines 2504-2507), projection lag becomes user-visible, or durable fan-out/backpressure becomes operationally important. EventStoreDB is the right next canonical store if aggregate stream consistency and persistent subscriptions matter; Postgres is the pragmatic next step if the team wants SQL operations and transactional outbox support; NATS JetStream is better as a fan-out bus than the canonical event store unless paired with a database.

The `EventStore` Protocol is not clean enough for a painless swap. `append/load/load_global/subscribe` (lines 195-224) lacks persistent subscription groups, ack/nack, leases, batch reads, backpressure, stream categories, expected-position failure types, and transactional command outbox semantics. The statement "No service code changes; only the binding" for EventStoreDB/Postgres/NATS (lines 857-864) is not credible. Define a second-level protocol now: `Subscription` with checkpoint/ack/error handling, `StoreCapabilities`, and explicit batch/catch-up APIs.

## 3. Reliability Gaps

The biggest reliability gap is atomic workflow progress. The spec says crashes mid-process are recoverable (line 36) and acceptance says killing the orchestrator resumes from `correlation_id` reconstruction (line 896), but correlation IDs are trace metadata, not durable state. For each process manager, persist "handled event", state mutation, and outgoing command(s) in one transaction. On restart, replay the outbox, not the correlation chain.

Projection checkpointing is improved by the `projection_checkpoints` table (lines 584-595), but the spec still must require atomic "apply event + update checkpoint" behavior. SQLite-backed projections can do this in one transaction. Chroma cannot, so `SemanticIndexProjection` (line 607) needs an idempotent side-effect ledger keyed by `ChunkId`, embedding model/version, and event `global_position`, plus a repair job for mismatches.

Idempotency is also underspecified. Unique `event_id` is necessary but only catches exact duplicates (line 691). Command de-dupe "stored in aggregate state" (line 692) will either grow unbounded or disappear incorrectly from snapshots. Use a bounded idempotency table keyed by `(aggregate_id, command_id)` with result event IDs. Content-addressed `Source`, `Chunk`, and `Artifact` IDs (line 693) must include source version, parser/chunker version, byte offsets, and normalization algorithm; otherwise a changed chunker can silently reuse old IDs.

Snapshot correctness needs more structure. The spec snapshots every 50 `Workspace` events and 200 `Index` events (line 277), but snapshot payloads need schema versions, upcasters, aggregate version, global position, and a hash of included event IDs. Otherwise a projection or aggregate replay can mix new event schemas with stale snapshot schemas.

## 4. Robustness Of The Citation Invariant

The current defense-in-depth version is much better than a plain `Cited[T].__post_init__`: Pydantic validators enforce non-empty citations, `EventStore.append()` revalidates hand-rolled dicts, and `WorkspaceProjection` constructs `Cited[T]` again (spec lines 541-568). That is the right architecture.

It is not airtight as written. Pydantic `model_construct()` bypasses validation; direct SQL imports can bypass `append()`; upcasters can create invalid payloads; and load paths can skip validation for performance. The spec's "There is no backdoor" claim (line 547) should be replaced with implementable rules: all event deserialization must use a registered Pydantic model via `model_validate`; `model_construct()` is banned outside tests; raw SQL event imports must run the same validator; and every upcaster output is validated against the next schema version before it can be yielded.

Non-empty citations are also only syntactic. The PRD requires every decision to cite evidence and lets verifiers reject uncited findings (PRD lines 1710-1722). The event model should validate that each `Citation.source_id` exists in `SourcesProjection`, has trust metadata, and points to an immutable source/chunk version. Add a canonical `NonEmptyCitations` type to the #8 schema package, not per-event ad hoc annotations, so `VariableLoaded`, `VariableEnriched`, `CitationAttached`, and `ToolInvoked` cannot drift (spec lines 504-529, 844-846).

## 5. Cohesion Across The 7 Issues

The overall issue integration is coherent: intake, parser, discovery, indexer, workspace, projections, and flows line up in the module layout (spec lines 141-180), and the ingestion-to-context flow is explicit (`ProjectCreated` -> parse/discover -> index -> workspace, lines 632-649). The hidden problem is that the five seams are not yet stable interfaces.

For intake -> parse, `PaperFetched` provides path and hash (lines 340-345), but parser inputs need blob URI, MIME/content validation result, paper version, and source trust. For parse -> index, the PRD expects equations, appendix text, paper tables, and source labels in the REPL (PRD lines 1219-1224), but the spec has section/reference/figure/table events and no equation or extraction-confidence event (spec lines 392-407). For discovery -> index, `ArtifactFound` is too raw (lines 435-438); add trust level, official/recommended status, license, commit/tag/date evidence, and contradiction hooks from the PRD source-priority model (PRD lines 1685-1695, 1851-1878).

For index -> load/workspace, the PRD has a unified Context REPL with raw variables plus `semantic_search()`, `graph_query()`, `rlm_query()`, `web_search()`, and `notebook_query()` (PRD lines 1167-1177), but the spec only commits to lookup and semantic search while deferring six tools (spec line 45). At minimum, define `ContextLoader` and `WorkspaceTool` Protocols, variable scope metadata, and the event contract for publishing enriched variables. The agent registry is also required by the PRD (PRD lines 241-260) but `WorkspaceReadyFlow` simply says "every agent registered to the project" (spec line 649); define the read-only registry interface even if the orchestrator implementation is out of scope.

## 6. Production-Grade Concerns Possibly Skimped

Security is the thinnest production section. The spec lists tokens in env vars, no `eval`/`exec`/`pickle`, TLS, and PDF size/time bounds (lines 733-738). The PRD threat model is much stronger: untrusted code, hidden network calls, credential exfiltration, package supply-chain attacks, poisoned datasets, and untrusted model weights (PRD lines 1820-1849). Even if runtime sandboxing is out of scope (spec lines 40-43), PDF parsing and embedding are in scope. Run PyMuPDF/Nougat parsing in an isolated worker with no network, tight memory/CPU limits, seccomp/AppArmor where available, and content-type sniffing. For embeddings, pin model package and weight digests, disable remote code, prefer safetensors, and ban implicit model downloads in production.

Captured HTTP headers are dangerous as specified because `request_headers` will include credentials unless redacted or encrypted (spec lines 299-306). Store a redacted header map by default, optionally encrypt sensitive values with a per-environment key, and never include bearer tokens in projection tables or dashboard events.

Observability is a good start but incomplete. Existing metrics cover events, command duration, projection lag, replay duration, and failed events (lines 711-720). Add outbox backlog, subscription retry count, dead-letter count, Chroma side-effect lag, blob missing/corrupt counts, HTTP capture failure count, circuit-breaker state, token/cost metrics for LLM/RLM calls, and security policy denials. Tests should add crash-injection around every append/outbox/checkpoint boundary; the current strategy covers happy replay and some invariants (lines 769-824) but not failure-point matrix testing.

Schema evolution also needs a real registry. The upcaster protocol exists (lines 279-292), but "register an upcaster" (line 683) must specify module discovery, chain ordering, renamed event types, golden fixtures per historical version, and snapshot upcasting. The #8 migration estimate of half a day (line 846) is unrealistic unless this registry is implemented first.

## 7. Anything To Kill Or Simplify

Do not event-source every process manager by default. The spec chooses event-sourced `flow_state` aggregates for consistency (lines 630, 876), but `IngestionFlow`, `IndexingFlow`, and `WorkspaceReadyFlow` are simple deterministic coordinators. A durable inbox/outbox plus a small state table per manager is easier to reason about and easier to make atomic. Reserve event-sourced process-manager state for workflows with user-visible decisions, approvals, or long-running negotiations.

`CapturedResponseFlow` should not be a process manager. The spec says it listens to external API calls and persists captured responses (lines 659-660), but deterministic replay depends on synchronous capture in the HTTP boundary itself. Make `shared/http.py` append `ExternalCallAttempted` before the call and `ExternalApiCalled`/`ExternalApiFailed` after the call. A later flow can compact blobs or audit gaps, but it should not be the primary recorder.

Snapshots are premature as fixed policy. "Every 50 events" for workspaces (line 277) adds correctness burden before replay latency is measured. Implement snapshot infrastructure, but trigger by measured replay time or event count thresholds from metrics. Also change `SemanticIndexProjection` rebuild semantics: re-embedding everything on rebuild (line 607) is expensive and not deterministic if model packages change. Rebuild Chroma from stored embedding blobs and only re-embed through an explicit reindex command.

## 8. Spec Gaps

Workspace scope policy is missing from the implementation spec. The PRD defines `private_to_parent`, `branch_shared`, and `global_verified` scopes (PRD lines 1078-1082) plus append-only shared-state rules (PRD lines 1384-1410). The spec's workspace events only have `workspace_id`, variable name, payload, citations, and agent fields (spec lines 499-529). Add `scope`, `task_id`, `parent_task_id`, `branch_id`, and promotion events.

Embedding backpressure is undefined. The spec chooses `all-MiniLM-L6-v2` by default (line 870) and emits `ChunkEmbedded` (lines 474-479), but it does not define queue depth, retry policy, rate limits, partial indexing status, model-version migration, or what `semantic_search()` returns while embeddings lag. Add `EmbeddingQueued`, `EmbeddingStarted`, `EmbeddingFailed`, and `EmbeddingCompleted` events or a durable embedding job table driven by events.

Blob storage layout is too thin. The only concrete layout is `runs/{project_id}/blobs/{sha}.bin` (line 305). Define atomic temp-file writes, fsync/rename, SHA verification on read, path normalization, refcounts or mark-and-sweep GC, encryption policy, maximum blob size by type, and cross-project dedupe rules.

Upcaster registration, projection checkpointing, and acceptance semantics need exactness. The upcaster protocol has no registry contract (lines 279-292). Checkpointing has a table but no batch size, transaction boundary, lock/lease model, dead-letter handling, or non-SQL projection rule (lines 584-617). The acceptance criteria are written as checked boxes before implementation (lines 887-900); make them unchecked requirements and add explicit tests for each. Finally, define the replay cache key for external calls and the redaction policy for captured request headers before implementation starts.
